I built an AI that does my homework — then emails 60 classmates

Building an unattended AI digest for a whole MBA cohort — and learning that for a system with no human in the loop, silence is a safer failure mode than a wrong answer.

Ujjwal Goel·ISB, FCRV course·Built with Claude

Every morning before our FCRV session — the ISB course that covers finance and macro — our professor forwards five to ten articles to the class: news clippings, research notes, the occasional screenshot. No consistent format, no consistent time. Some mornings it's three PDFs at 7am; other mornings it's nine at 11pm the night before. Reading all of it, every day, on top of actual coursework, was not sustainable.

So I built an agent. It watches a synced folder, notices new PDFs as they land, summarizes each one in a consistent teaching-oriented format, and emails a clean digest to the whole class — every morning, before I've woken up. No one has to remember to run it. That last part is what makes this a genuinely interesting system to build, and the part I want to write about: not "I used Claude to summarize a PDF," but everything that had to be true for an unattended system to be trustworthy enough to email real people every day without me watching it.

The naive version, and why it breaks immediately

The first version I sketched was embarrassingly simple: each morning, look at what's in the folder, summarize everything in it, send the digest. It took about an hour to see why that doesn't survive contact with reality.

If the job reruns daily and just "summarizes what's in the folder," it re-summarizes everything it already covered yesterday, plus whatever's new — duplicate content, growing runtime, and eventually a digest nobody trusts because half of it is repeats. The PDFs don't even arrive directly; they land via a separate pipeline (professor's email → OneDrive sync), which isn't instant — so a naive job can start reading a folder that's only half-synced and miss articles entirely. Some PDFs are clean text; others are scanned images with no extractable text at all, which a plain text-extraction step just returns empty for. And worst of all: if any single step breaks — one unreadable file, one bad response — the naive version has no fallback. It either crashes (no digest, fine) or, worse, keeps going and sends something broken to sixty people who have no way to know it's wrong.

The manifest is the single source of truth

Everything downstream depends on one small JSON file — _covered.json — that lists every file the agent has already summarized. Each run reads it once, diffs it against what's actually in the folder, and only touches genuinely new files. It never re-opens anything already covered, even "just to double-check." That restraint matters: the moment you let a system re-read something for verification, you've reintroduced the reprocessing problem you built the manifest to avoid.

Waiting out the sync, on purpose

The PDFs arrive through a pipeline I don't control: the professor's email triggers a OneDrive sync, and that sync takes a few minutes, not zero. So the agent has a guard step before it ever looks at the folder: if the professor's email landed in the last fifteen minutes, wait. Enumerating a half-synced folder is worse than enumerating a stale one — a stale folder just means "try again next run," but a half-synced folder looks complete and lets the agent quietly skip articles that hadn't arrived yet.

Fail silent, fail safe

This is the design decision I keep coming back to. If any step breaks — an unreadable PDF, a failed lookup, anything — the agent produces no output, rather than a broken or partial one. It never invents content to fill a gap. The next day's run just retries, because the manifest was never updated for the file that failed, so it's still "new" tomorrow.

It would have been easy to make the opposite choice — always send something, since a partial digest feels more useful than none. But this system emails an entire class automatically, with nobody checking it before it goes out. A wrong or half-built summary erodes trust in a way that's hard to earn back, and it does so silently, because nobody's proofreading it before it lands in sixty inboxes. A missing digest is an obviously visible, easily forgiven gap. A wrong one looks authoritative. For any system that acts on real people without a human in the loop, I'd treat "silence is the safe default" as a starting assumption, not an edge case to patch in later.

Self-healing catch-up

Because the manifest — not the calendar — decides what's new, a missed run (a holiday, a bug, me forgetting to check on it) isn't a lost day. The next run doesn't ask "what came in today?" It asks "what's in this folder that isn't in the manifest yet?" — regardless of how old the file is. A three-day backlog gets absorbed into the next digest exactly the same way one new file would. The system doesn't need a separate backfill mode; catch-up is just the normal path, running on a folder with more in it than usual.

One format, no matter how messy the input

The inputs are wildly inconsistent — a Bloomberg clipping, a screenshot of a WhatsApp forward, a dense research note — but every summary is forced into exactly two blocks. First, a plain-English causal-chain narrative: news → underlying cause → visible effect → market read-through, written with no bullet points and no jargon. Second, a "Key Takeaway" that bridges the article to a specific concept from that week's FCRV session — risk premium, cost of debt, opportunity cost — sometimes weaving in the professor's own commentary pulled from his email. The constraint is deliberate: heterogeneous source material, pushed through the same fixed cognitive scaffold every time, so a reader builds the same mental model on day thirty as on day one.

The agent never sends email directly — it only has to produce a correctly-named, well-formed file. A separate system watches for it and acts. That decoupling is one of the more reusable ideas here.

What classmates actually said

I ran a short anonymous survey after a few weeks of daily digests. Twelve responses came back, with an average helpfulness rating of 4.1 out of 5.

"I actually started reading the articles sent by prof more after receiving your emails, thinking have I become so lazy. Thank you for making me more diligent xD"
"It was a really great initiative you took. It was genuinely helpful."

What I'd carry into the next automation

Three ideas here weren't specific to summarizing PDFs, and I'd reach for them again on any unattended system:

A manifest beats a calendar. "What have I already done?" is a more reliable question than "what should I do today?" — it makes catch-up free and reprocessing impossible, with no separate backfill logic required.

Silence is a valid, often correct, output. Any system that acts on real people without a human checking first should default to doing nothing on uncertainty, not to doing its best guess.

Decouple the action from the artifact. The agent's only job is to produce a well-formed file with the right name. A separate system decides what to do when that file appears. That seam made the whole thing easier to reason about, and easier to replace either half of later.
